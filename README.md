## 这本书讲什么

这本书讨论的是自动驾驶场景中的实时中间件。它关注的不是企业级消息队列、应用服务器或传统业务中间件，而是运行在车端、服务于实时 / 嵌入式分布式系统的那一层基础设施：它要让传感器、感知、融合、规划、控制、执行器、诊断工具和系统服务在有限时间、有限资源和复杂故障条件下协同工作。

全书的核心问题可以概括为一句话：自动驾驶中间件为什么会从早期的通信管道，逐渐长成近似分布式实时操作系统的形态？回答这个问题，不仅要看某一个具体框架，也不能只比较谁更快、谁更流行。更重要的是理解每一代方案背后的约束：确定性、效率和通用性很难同时满足；需求、算法、硬件、规模化和安全合规又会不断改变最优点。中间件的演进，正是在这些约束之间反复重新定位。

全书按五个部分展开。第一部分回答“为什么需要中间件，以及它逃不掉哪些约束”：从中间件在自动驾驶软件栈中的位置讲起，再用一辆小车从相机到刹车执行器的链路说明没有中间件会遇到哪些问题，最后把这些问题收束为全书的理论地图。第二部分讨论中间件的三层能力：通信传输层解决数据如何从 A 到 B，调度执行层解决代码何时、在哪里运行，系统服务层解决整机如何启停、切换、监控、降级和部署。第三部分转向横切能力，讨论数据表达、时间、可观测与回放、安全与可控如何穿过多层，并如何在效率、确定性和通用性之间继续产生代价。第四部分把前面的机制放回端到端账本，用时延、确定性、资源容量和可扩展性解释为什么局部优化不一定等于整车正确。第五部分回到全局视角，讨论分层带来的跨层张力，重读中间件演进史，对照真实系统的取舍，并展望自动驾驶中间件继续向分布式实时操作系统演进的方向。

预期读者有三类。刚接触自动驾驶软件的人，可以把它当作一张系统地图，先理解中间件在哪里、为什么需要、主要解决哪些问题；已经在工程现场使用 ROS 2、Cyber RT、AUTOSAR AP、DDS、Zenoh 或类似系统的人，可以把它当作一套权衡框架，用来判断不同方案的边界和代价；对实时系统、嵌入式系统或自动驾驶平台已有经验的读者，则可以重点关注端到端确定性、跨层信息屏障、零拷贝与可观测性、安全与迭代之间的结构性张力。

这本书不试图给出一个“唯一正确的中间件选型答案”。它更希望提供一把可复用的尺子：当读者面对一个具体系统时，能够看清哪些能力属于中间件边界，哪些问题必须回到组件内部解决；哪些优化只是局部收益，哪些代价会在端到端链路上重新出现；以及为什么今天看似合理的架构，可能会在下一代算法、硬件或安全要求到来时重新漂移。
